# Q-BU 拡張性設計（B2B拡張の土台）

> 方針：**v1は最小実装**。でも「あとから足せる」**接ぎ目（seam）だけ先に用意**しておく。
> 機能そのものは作らない。**土台だけ**残す。

---

## 0. 結論：3つの土台で、ほぼ全機能が「乗せるだけ」になる

| 土台 | 何のため | 今やること |
|---|---|---|
| ① 前方互換データ形式 | 後から項目を足してもデータが壊れない | `schema` 付きJSON envelope |
| ② 機能フラグ + 設定（CONFIG） | ホワイトラベル・機能ゲート（**B2Bの核**） | `CONFIG` を1個置く |
| ③ ボクセル→メッシュ共有パイプライン | 編集・STL・AR・サムネを1つの生成器に | 生成器を1本化する設計 |

> データ形式は **後から直すのが一番つらい**。だからここだけは今きっちり前方互換にしておく。

---

## 2. データ形式（前方互換スキーマ）

```jsonc
{
  "schema": "qbu.creation/1",        // ← バージョン。読む側が判定・移行できる
  "meta": {
    "appVersion": "0.1.0",
    "tenant": "default",             // B2B: 顧客/組織
    "event":  "default",             // ワークショップ/イベント単位
    "mode":   "model",               // 将来: paint / code ...
    "createdAt": "ISO8601"
  },
  "creator": { "id": "A-0427", "name": "" },
  "model": {
    "unit":   1,                     // 1ブロックの基準サイズ
    "bounds": [100, 100, 100],
    "blocks": [
      { "x":0, "y":0, "z":0, "color":"#9AA0A6" }   // 将来: material, texture, shape
    ]
  },
  "ext": {}                          // 将来機能の自由枠（実験はまずここ）
}
```

### 前方互換の4ルール（これだけ守る）
1. **追加のみ** — 既存フィールドの意味を変えない・消さない
2. **未知フィールドは保持** — ホストは受信JSONを**そのまま保存**（古い版が新項目を消さない）
3. **`ext` は自由枠** — 正式化前の実験フィールドはここに入れる
4. **見た目はブロックが持つ** — 色/素材/形はブロック単位の属性として足す

### どの機能がどこに乗るか
- 複数色 → `block.color`（もう枠がある）
- テクスチャ → `block.material` / `block.texture` を追加
- 形状追加 → `block.shape`
- モード → `meta.mode`
- マルチテナント → `meta.tenant` / `meta.event`
- 投稿 → `meta` ＋ 保存先（公開フラグ等は `ext` から昇格）
- **AR・STL・サムネ** → `model` をそのまま読んで生成（③で1本化）

## 3. コードの拡張ポイント（seam）

| seam | 役割 | 今の状態 |
|---|---|---|
| **CONFIG** | 機能フラグ・パレット・上限・ブランディング・テナント | mock.html に設置済み |
| データ envelope | 前方互換の送信形式 | mock/host に実装済み |
| ボクセル→メッシュ生成器 | 編集表示が、そのまま STL/GLB/AR/PNG の元になる | 生成は1箇所（将来ここから書き出し） |
| モード/ツール登録 | model/paint/code… を足せる仕組み | 将来：レジストリ化 |
| マテリアル抽象化 | ブロックが見た目を持てる | block.color の枠あり |
| exporter 登録 | STL/GLB/USDZ/PNG を差し替え可能に | 将来：プラグイン化 |
| テーマ = CSS変数 | ホワイトラベル | 既に変数化済み（`--bg` 等） |
| identity 差し込み口 | 自動ID＋合言葉 → 本認証へ | getId() を後で差し替え |

## 4. 段階ロードマップ（順番の目安・実装はまだしない）

- **v1（今）**：最小。ブロック並べる→送信→完了。単色。土台だけ仕込む。
- **v1.1**：複数色（palette増やすだけ）、Undo、範囲削除
- **v1.2**：GLB書き出し → **AR表示**（model-viewerが一番早い）
- **v1.3**：投稿 / ギャラリー
- **v2（B2B本格）**：マルチテナント / ホワイトラベル / 機能ゲート（課金）/ 認証・ロール / 解析

---

## 5. 今回コードに入れた「元」（実装ではなく土台）

- `mock.html`：`CONFIG`（機能フラグ・パレット・テナント・上限）を設置
- `mock.html`：送信を**前方互換 envelope** に変更（`schema`/`meta`/`creator`/`model`/`ext`）
- `host.py`：受信JSONを**そのまま保存**＝未知フィールドを保持（前方互換の肝）

> どれも「機能」ではなく「あとで機能を足すための接ぎ目」。フラグは今は最小。
